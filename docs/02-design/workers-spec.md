## 1. 기능 개요 및 목적

### 1.1 개요

작업자 관리 화면(`/master/workers`)은 임진강김치(주) 생산 현장에서 근무하는 작업자의 기준정보를 등록·조회·수정·삭제하는 기준정보 관리 페이지입니다. 기존 `/master/customers` 페이지와 동일한 UI 패턴(PageHeader + 검색/필터 바 + Table 컴포넌트)을 따릅니다.

### 1.2 목적

| 목적 | 설명 |
|------|------|
| 생산인력 현황 파악 | 공정별·교대조별 배치 인원을 실시간으로 확인 |
| 작업지시 연동 기반 | MES 작업지시 발행 시 작업자 코드를 참조하여 담당자 자동 배정 |
| 인사정보 단일 출처 | 재직/휴직/퇴직 상태를 한 곳에서 관리하여 생산기록에 일관성 확보 |
| 교대조 편성 지원 | 1조/2조/3조 편성 현황을 시각적으로 확인하여 균형 배치 지원 |

## 2. 사용자 스토리

### 2.1 생산관리자 관점

| # | 스토리 | 수용 기준 |
|---|--------|-----------|
| US-01 | 공정별 배치 인원을 한눈에 파악하고 싶다 | 공정 필터 선택 시 해당 공정 작업자 목록만 표시되어야 한다 |
| US-02 | 특정 작업자의 담당 공정과 교대조를 빠르게 조회하고 싶다 | 이름 또는 사원번호로 검색 시 0.5초 이내 결과가 표시되어야 한다 |
| US-03 | 작업지시 등록 시 해당 공정의 재직 작업자 목록을 선택할 수 있어야 한다 | 상태=재직인 작업자만 작업지시 담당자 선택 목록에 노출되어야 한다 |

### 2.2 현장반장 관점

| # | 스토리 | 수용 기준 |
|---|--------|-----------|
| US-04 | 내 교대조(예: 1조) 작업자 명단을 즉시 확인하고 싶다 | 교대조 필터 선택 시 해당 조 인원만 필터링되어야 한다 |
| US-05 | 작업자가 결근 등으로 임시 공정 이동 시 담당공정을 수정하고 싶다 | 수정 모달에서 담당공정 변경 저장 후 목록이 즉시 반영되어야 한다 |
| US-06 | 신규 현장인력 투입 시 빠르게 등록하고 싶다 | 필수 항목(사원번호, 이름, 부서, 교대조, 상태) 입력 후 저장이 3초 이내에 완료되어야 한다 |

### 2.3 인사담당자 관점

| # | 스토리 | 수용 기준 |
|---|--------|-----------|
| US-07 | 퇴직자 처리 시 시스템 상태를 '퇴직'으로 변경하고 싶다 | 퇴직 처리 후 해당 작업자는 작업지시 담당자 목록에서 제외되어야 한다 |
| US-08 | 전체 작업자 현황을 Excel로 내보내고 싶다 | 목록 화면에 Excel 내보내기 버튼이 제공되어야 한다 (Should) |
| US-09 | 입사일, 재직기간 등 인사 기본 정보를 조회하고 싶다 | 작업자 상세 보기 모달에서 입사일 및 비고 항목이 표시되어야 한다 |

## 3. 데이터 모델

### 3.1 workers 테이블

| 필드명 | 타입 | 길이 | 필수 | 설명 | 예시 |
|--------|------|------|------|------|------|
| `id` | INTEGER | - | 자동 | PK, 자동증가 | 1 |
| `worker_code` | VARCHAR | 20 | 필수 | 사원번호 (유니크) | `WRK-001` |
| `worker_name` | VARCHAR | 50 | 필수 | 작업자 이름 | 김절임 |
| `department` | VARCHAR | 50 | 필수 | 부서 (공정 단위) | 절임공정팀 |
| `position` | VARCHAR | 30 | 선택 | 직급 | 반장 / 선임 / 작업원 |
| `process_type` | VARCHAR | 30 | 필수 | 담당공정 | 세척 / 절임 / 양념 / 포장 / 품질검사 / 출하 |
| `shift` | VARCHAR | 10 | 필수 | 교대조 | 1조 / 2조 / 3조 |
| `phone` | VARCHAR | 20 | 선택 | 연락처 (휴대전화) | 010-1234-5678 |
| `hire_date` | DATE | - | 선택 | 입사일 | 2022-03-15 |
| `status` | VARCHAR | 10 | 필수 | 재직 상태 | 재직 / 휴직 / 퇴직 |
| `note` | TEXT | - | 선택 | 비고 (특이사항) | 위생교육 이수 완료 |
| `is_active` | BOOLEAN | - | 자동 | 활성 여부 (status 연동) | true / false |
| `created_at` | TIMESTAMP | - | 자동 | 등록일시 | 2026-01-10T09:00:00 |
| `updated_at` | TIMESTAMP | - | 자동 | 최종수정일시 | 2026-05-13T14:30:00 |

| `salting` | 절임 | 소금 절임 공정 |
| `seasoning` | 양념 | 양념 배합 및 버무리기 |

### 4.3 작업자 삭제

- 재직 상태의 작업자는 직접 삭제 불가. "퇴직 처리" 후 논리 삭제(is_active=false) 방식 적용
- 퇴직/휴직 상태인 경우 확인 다이얼로그 후 논리 삭제 허용
- 삭제 확인 메시지: "해당 작업자를 삭제하면 관련 작업 이력은 보존됩니다. 삭제하시겠습니까?"

## 5. 비즈니스 규칙

### 5.1 교대조 관리 규칙

| 규칙 | 내용 |
|------|------|
| BR-01 | 1조/2조/3조의 3교대 근무제로 운영하며, 각 작업자는 반드시 하나의 교대조에 소속되어야 한다 |
| BR-02 | 교대조 변경은 수정 모달에서 즉시 반영되며, 변경 이력은 시스템 로그에 기록된다 |
| BR-03 | 동일 교대조 내 동일 공정에 최소 1명 이상 배정되어야 하며, 미충족 시 경고 알림을 표시한다 |

### 5.2 공정별 최소 인원 기준

| 공정 | 최소 인원 (전체) | 교대조당 최소 |
|------|---------------|--------------|
| 세척 | 4명 | 1명 |
| 절임 | 6명 | 2명 |
| 양념 | 6명 | 2명 |
| 포장 | 6명 | 2명 |
| 품질검사 | 2명 | 1명 |
| 출하 | 3명 | 1명 |

- 최소 인원 미달 시 공정 현황 화면에 경고(빨간색 배지) 표시
- 최소 인원은 시스템 관리자가 설정값으로 변경 가능 (Should)

### 5.3 퇴직 시 처리 규칙

| 단계 | 처리 내용 |
|------|-----------|
| BR-04 | 작업자 상태를 '퇴직'으로 변경하면 `is_active = false` 로 자동 설정 |
| BR-05 | 퇴직 처리된 작업자는 작업지시 담당자 선택 목록에서 자동 제외 |
| BR-06 | 퇴직 처리된 작업자의 과거 작업 실적 데이터는 보존 (논리 삭제, 물리 삭제 금지) |
| BR-07 | 퇴직 처리 시 해당 작업자가 배정된 미완료 작업지시가 있을 경우, 재배정 요청 팝업을 표시 |
| BR-08 | 물리 삭제(DB DELETE)는 슈퍼관리자만 허용하며, 별도 감사 로그 기록 |

## 7. 작업 우선순위 및 예상 개발 공수

### 7.1 MoSCoW 우선순위

| 우선순위 | 기능 | 근거 |
|---------|------|------|
| Must | 작업자 목록 조회 (검색, 공정/교대조/상태 필터) | MES 핵심 기준정보 |
| Must | 작업자 등록/수정 (모달 폼) | 데이터 입력 필수 |
| Must | 퇴직 처리 (논리 삭제) | 작업지시 연동 신뢰성 |
| Must | 백엔드 CRUD API (GET/POST/PUT/DELETE) | 프론트 연동 전제 |
| Should | 공정별 배정 현황 요약 테이블 | 운영 편의성 |
| Should | 최소 인원 미달 경고 표시 | 생산 안정성 |
| Should | Excel 내보내기 | 인사팀 요청 |
| Could | 작업자 변경 이력 조회 | 감사 추적 |
| Could | 공정 이동 일괄 처리 | 교대조 개편 시 필요 |
| Won't | SSO 연동 (HR 시스템 자동 동기화) | 향후 Phase에서 검토 |

### 7.2 예상 개발 공수

| 작업 | 담당 | 공수 (인일) | 비고 |
|------|------|------------|------|
| DB 테이블 설계 및 마이그레이션 | 백엔드 | 0.5일 | `workers` 테이블 DDL |
| FastAPI CRUD 엔드포인트 구현 | 백엔드 | 1.5일 | 목록/단건/등록/수정/삭제 |
| 공정별 현황 집계 API | 백엔드 | 0.5일 | GROUP BY 쿼리 |
| Pydantic 스키마 정의 | 백엔드 | 0.5일 | Request/Response 모델 |
